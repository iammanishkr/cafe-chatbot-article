01 — THE PROBLEM

Picture this: you're a tourist in Bangkok. You walk into a cozy café. The menu is in Thai. The staff don't speak English. You open Google Translate, spend five minutes fumbling around, point at something random on the screen, and end up with a bowl of something warm that might be soup.

Now flip the scenario. You're a café owner in Pune. Tourists from Thailand, Korea, and Germany visit every week. Your staff can't communicate with half of them. Orders get lost. Customers leave frustrated. Revenue walks out the door with them.

The world is multilingual. Most software isn't. That gap is a product opportunity — and this project is one answer to it.

02 — THE SYSTEM

The Multilingual Smart Chatbot That Actually Understands You

This project — built during a six-month internship at Vebsigns Technologies — is a full-stack AI system that handles café orders through natural conversation. No forms. No dropdowns. Just talk to it — in English, Hindi, or Thai — and it figures out what you want, queries the menu, calculates your total, and saves your order.

It supports both text and voice input. It runs three transformer models in a coordinated pipeline. It can handle 120 requests per second. And it does all of this with sub-500ms latency.

✦ Live scenario

You type: "मुझे दो कैपुचीनो और एक चॉकलेट मफिन चाहिए"

The chatbot replies in Hindi, confirms your order, shows you the total (₹227), and records it in the database — all in under 400 milliseconds.

04 — THE INTELLIGENCE

The Part That Makes It Actually Smart: A Three-Model NLP Pipeline

Here's where most people's eyes light up. This isn't a chatbot that checks your message against a list of keywords and gives you a canned response. It runs three specialized transformer models in sequence — each one handling a specific cognitive task.

Think of it like a kitchen brigade: the translator, the intent reader, and the backup chef who steps in when things get ambiguous.

1
facebook/nllb-200-distilled-600M
🌐 Cross-Lingual Translation

Meta's "No Language Left Behind" model translates the user's input into English before any analysis happens. It uses beam search decoding with INT8 quantization — cutting inference time by ~42% without a meaningful drop in quality. It supports 200 languages, so future expansion is trivial.

2
xlm-roberta-large-xnli
🎯 Intent Classification

XLM-RoBERTa (fine-tuned on the cross-lingual NLI corpus) reads the translated text and assigns one of nine intents. It treats each intent as a hypothesis and evaluates whether the user's message entails it. Accuracy sits at 94.3% across all three languages. If confidence drops below 0.75, it passes the baton.

3
paraphrase-multilingual-MiniLM-L12-v2
🔍 Semantic Similarity Fallback

When classification isn't confident, Multilingual MiniLM encodes the input as a 384-dimensional dense vector and finds the closest match among pre-computed template embeddings using cosine similarity. It's the graceful safety net that handles unusual phrasing, slang, and edge cases.

05 — THE ENGINEERING

Async All the Way Down — and Why It Matters

One of the best technical decisions in this project is going fully asynchronous from day one. FastAPI + Uvicorn (ASGI) means the system handles concurrent requests without blocking — no user waits for someone else's database query to finish.

# Parallel async menu fetch — both queries run simultaneously
menu_items, featured_items = await asyncio.gather(
    db_helper.fetch_menu_items(
        category=req.category,
        language_code=req.language
    ),
    db_helper.fetch_featured_items(req.language)
)

# Result: two database calls in the time it takes to run one

The backend also implements a smart stateless-but-context-aware pattern. Rather than keeping session state on the server, the client sends a session_id with every request. The server looks up cart state from the database using that token.

💡

This design means no sticky sessions for load balancing, no synchronization headaches across servers, and trivial horizontal scaling. Small decision — massive payoff at scale.

What Happens in 400ms: A Message's Journey

Let's trace a single message — "मुझे दो कैपुचीनो चाहिए" — from the user's keyboard to a confirmed order in the database.

1️⃣
Frontend sends POST to /handle-message with the message, language: "hi", and a session ID
2️⃣
Pydantic validates the payload. Invalid input fails fast with a 422 and helpful error detail.
3️⃣
NLLB translates Hindi → English: "I want two cappuccinos"
4️⃣
XLM-RoBERTa classifies intent as add_item with high confidence
5️⃣
Backend parses item (cappuccino) and quantity (2), queries MySQL for price
6️⃣
Cart updated via parameterized SQL with ON DUPLICATE KEY UPDATE — safe, atomic, idempotent
7️⃣
Localized Hindi response returned with order confirmation and updated cart total

08 — THE LESSONS

What's Worth Stealing From This Project

  • 1
    Use a pipeline, not a monolith. Three specialized models beat one general model trying to do everything. Delegation is not a weakness — it's architecture.
  • 2
    Go async from day one. Retrofitting async into synchronous Python code is painful and error-prone. Start right. Your future self (and your users) will thank you.
  • 3
    Localize everything — including errors. An error message in the wrong language destroys trust instantly. If you support multilingual users, your errors need to be multilingual too.
  • 4
    Stateless + session tokens = scalable. Don't store session state on the server if you can avoid it. Pass a token, look up context, stay horizontally scalable.
  • 5
    Fallback mechanisms are not optional. Real users say unexpected things. Plan for ambiguity with layered fallbacks — not just an error message, but an actual attempt to help.
  • 6
    Modular architecture compounds over time. Every feature listed in the roadmap is addable without rebuilding the core. Design for extension from the start.
